# PetPulse Database Schema Explanation

This document serves as a comprehensive breakdown of the PetPulse PostgreSQL database. You can use this guide to confidently explain the database architecture, tables, and columns to your professors or colleagues.

---

## 1. Core Users and Profiles

### `users`
This is the foundational table for the entire platform. Every individual (pet owner, vet, trainer, or admin) registers here.
* **`id`**: Unique UUID for the user.
* **`email` / `password_hash`**: Secure login credentials.
* **`first_name` / `last_name`**: User's real name.
* **`role`**: ENUM indicating if they are an `owner`, `vet`, `trainer`, or `admin`.
* **`latitude` / `longitude`**: Standard float values used to track the user's geographic location (crucial for finding nearby vets or lost pets).
* **`push_token`**: Used to send mobile push notifications.

### `vet_profiles` & `trainer_profiles`
These tables extend the `users` table. If a user is a Vet or a Trainer, they get an additional record here.
* **`user_id`**: Links directly back to the `users` table.
* **`clinic_name` / `license_number`**: Specific verification details for Vets.
* **`specialties`**: Array of text describing a trainer's specific skills.
* **`status`**: ENUM (`pending`, `approved`, `rejected`) allowing admins to verify professionals before they appear on the app.

---

## 2. Pet Management

### `pets`
Stores the profiles of all pets registered by users.
* **`id`**: Unique identifier for the pet.
* **`owner_id`**: Foreign key linking the pet to their owner in the `users` table.
* **`name`, `species`, `breed`**: Basic identifying information.
* **`age_years`, `weight_kg`**: Health and physical metrics.
* **`avatar_url`**: Link to the pet's profile picture.

---

## 3. Booking and Appointments

### `appointments` (Veterinary Bookings)
Handles bookings made between Pet Owners and Veterinarians.
* **`pet_id`**: The specific pet needing care.
* **`vet_user_id`**: The doctor assigned to the appointment.
* **`status`**: Current state (`pending`, `confirmed`, `completed`, `cancelled`).
* **`appointment_time` / `reason`**: When and why the appointment is happening.
* **`handled_by_ai`**: A boolean flag indicating if this appointment was fully automated and scheduled by our Agentic AI rather than human intervention.

### `services` & `service_bookings` (Marketplace)
Allows Trainers and Sitters to offer paid services.
* **`services` table**: Defines the offering (e.g., "Dog Walking", Category: `walking`, `base_price`).
* **`service_bookings` table**: Records the actual transaction when a user books a service, tracking `start_time`, `end_time`, `total_price`, and whether the booking was `handled_by_ai`.

### `payments`
Tracks financial transactions securely.
* **`booking_id`**: Links the payment to a specific service booking.
* **`payer_id` / `payee_id`**: Links the flow of money between two users.
* **`amount` / `currency`**: The financial value.
* **`gateway_transaction_id` / `gateway_name`**: Integration points for third-party gateways (like Paymob or Stripe).

---

## 4. Lost & Found System

### `lost_pets`
Triggered when an owner reports their pet missing.
* **`pet_id`**: The pet that is lost.
* **`latitude` / `longitude`**: Exact geographic coordinates where the pet was last seen.
* **`lost_time` / `description`**: Context to help finders identify the pet.

### `found_reports`
Triggered when a user spots a stray or lost pet.
* **`reporter_id`**: The user who found the pet.
* **`lost_pet_id`**: (Optional) If the system successfully matches this report to an existing `lost_pets` record.
* **`latitude` / `longitude`**: Where the pet was found.
* **`image_url`**: Photo taken by the finder.
* **`ai_match_score`**: A decimal score (0.0000 to 1.0000) generated by our AI image recognition to calculate the probability that the found pet matches a reported lost pet.

## 5. Community and AI Integration

### `community_posts`
A lightweight table to support basic social interactions.
* **`user_id`**: The author of the post.
* **`content`**: The text of the post.
* **`likes_count`**: Basic engagement metric.

### `ai_booking_sessions`
The backbone of the platform's core Agentic AI feature. This table gives the AI "memory" during automated negotiations.
* **`user_id`**: The user interacting with the AI.
* **`target_service_id` / `target_vet_id`**: The specific service or doctor the AI is trying to book on behalf of the user.
* **`status`**: The state of the AI conversation (`active`, `completed`, `failed`).
* **`conversation_history`**: A structured `JSONB` column that stores the chat log between the user and the AI, giving the LLM context so it remembers the conversation flow (e.g., "I want an appointment tomorrow" -> "How about 3 PM?").
